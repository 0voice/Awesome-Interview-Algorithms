### 字节跳动
### 出题偏好
- **整体偏好**：
字节跳动算法面试/笔试以「LeetCode 高频题 + 推荐/AI 业务深度融合」为核心，既考察基础算法的工程实现与优化能力，更注重算法在短视频分发、内容推荐、实时计算等核心场景的落地适配（如视频排序、用户行为预测）。笔试时长通常 2 小时，题量因岗位差异显著：算法岗为「选择题 + 4-5 道编程题」，后端岗为「选择题 + 3 道编程题」，客户端/数据岗为「选择题 + 2-3 道编程题」，整体难度呈递进式分布（如前 1-2 题基础 Easy，最后 1 题达 Medium-Hard 难度）；技术一面聚焦「算法正确性与边界优化」，要求手撕代码并分析海量数据下的复杂度（如 O(n) 时间 + O(1) 空间的数组问题），二面侧重「AI/ML 融合场景」，三面引入「系统扩展与指标量化」（如算法如何提升视频完播率或降低推荐延迟）。
  - **高频主题占比**：动态规划（DP）（24%）、贪心（22%）、树/图（20%）、数组/字符串（18%）、其他（搜索/堆/ML 基础）（16%）。
  - **趋势**：大模型与推荐融合题占比激增（如基于 Transformer 的序列 DP 优化、用 RAG 提升内容覆盖率），AI 工程化考察强化（如模型部署的并发适配、隐私保护下的图算法），但经典基础（如 DP 状态转移、贪心证明正确性）的业务验证仍是评判底线。
- **岗位偏好：**
  - 后端（e.g., 内容分发系统、飞书协作、抖音云）：DP/贪心/图（39%），变体多为「高并发优化问题」（如带缓存的视频路径规划变体），高频题包括「区间 DP 的多约束适配」「子数组和的滑动窗口」等，业务场景如内容 CDN 调度、分布式日志的负载均衡、实时流数据的异常检测。
  - 算法岗（推荐/广告/搜索）：DP/贪心/ML 融合（44%），侧重「用户行为驱动的模型设计」（如基于观看轨迹的字符串 DP 推荐变体），融合特征工程（如视频互动的时间/语义特征融入贪心状态），常考察「算法与 KPI 的量化关联」（如如何用模型提升用户留存率或降低广告跳出）。
  - 客户端（e.g., 抖音 App、头条客户端）：数组/字符串/链表（34%），偏「高效端侧实现」（如 O(1) 空间的本地内容解析），高频题涵盖「数组双指针的滑动匹配」「链表的条件反转」等，业务场景如 App 视频预加载的增量同步、用户标签的快速检索、滑动界面性能调优。
  - 数据开发/数仓：图/DP/并行计算（41%），核心考察「TB 级内容的处理鲁棒性」（如亿级视频日志的图谱分析变体），适配 Flink/Spark 框架，业务场景如跨域内容数据的实时聚合、用户偏好轨迹的离线建模、广告投放数据的去重与 A/B 测试分析。

阿里巴巴算法面试/笔试以「LeetCode 企业高频题 + 电商/搜索业务场景适配」为核心，既考察经典数据结构与算法的实现与优化（如时间/空间复杂度分析），更注重算法在淘宝/天猫推荐、搜索排序、实时风控等核心业务中的应用（如商品排序、用户画像预测）。笔试时长通常 120 分钟，题量因岗位差异明显：算法岗为「15 道选择题 + 3-4 道编程题」，后端岗为「15 道选择题 + 3 道编程题」，客户端/数据岗为「15 道选择题 + 2-3 道编程题」，难度渐进（前 1-2 题 Easy-Medium，最后 1 题 Medium-Hard）；技术一面强调「代码正确性与边界处理」，要求手撕代码并讨论海量数据下的优化（如 O(n log n) 排序在亿级商品中的适用性），二面聚焦「业务融合与扩展」（如算法如何提升搜索召回率），三面引入「系统设计与量化评估」（如在分布式环境下的图算法部署对延迟的影响）。
  - **高频主题占比**：数组/字符串（32%）、动态规划（DP）（28%）、哈希表/贪心（18%）、树/图（15%）、其他（堆/并查集/二分）（7%）。
  - **趋势**：大模型与搜索推荐融合题占比上升（如基于 BERT 的字符串匹配优化、用 GNN 提升商品图谱覆盖），AI 工程化考察增多（如模型推理的并行适配、联邦学习下的隐私图算法），但经典基础（如 DP 状态压缩、贪心证明、树遍历变体）的业务验证仍是核心底线。

  - 后端（e.g., 淘宝、天猫、阿里云）：数组/DP/哈希表（45%），变体多为「多约束条件」（如带权重的背包变体），业务场景如订单排序、库存调度、分布式缓存优化。
